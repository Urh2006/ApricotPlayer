# ApricotPlayer v0.9.44 Beta 5

Fifth beta in the 0.9.44 cycle. Addresses tester feedback after beta.4: audio quality regression (perceived stereo narrower than before the modular refactor), slow song start time, and high RAM use during extended listening sessions.

## Bug fixes

- **Audio quality restored to pre-refactor levels.** `resolve_stream_url` had a hardcoded format selector `"best[ext=mp4]/best"` that always picked a combined 720p MP4 stream from YouTube. These progressive MP4 streams carry AAC audio at 128 kbps. The correct selector respects the user-configured format setting and, for background playback, prefers an audio-only stream first: `bestaudio[ext=m4a]/bestaudio/…`. On YouTube this yields M4A at 256 kbps or Opus at 160 kbps — noticeably better stereo separation and dynamic range than the progressive AAC stream. The fallback chain still covers the user's configured video format and known-safe format IDs (18, 22) for servers that don't expose audio-only tracks.

- **Player starts faster.** The root cause is the same as the quality regression: when mpv plays a YouTube song with `--vid=no` (audio-only mode), it was previously fetching the combined 720p MP4 stream — 2–4 Mbit/s of video data it immediately discards. With an audio-only stream (~160 kbps), mpv buffers far less before it can begin playback and the apparent start lag drops dramatically. Seeking is also faster.

- **Multi-track DASH stream URLs are now extracted correctly.** When `bestvideo+bestaudio` is selected (e.g. user set Format to "Best available"), the yt-dlp info dict does not populate `info["url"]`; instead the individual streams are in `info["requested_formats"]`. The fallback now walks `requested_formats` — preferring the audio track — before scanning the full format list.

## Performance improvements

- **RAM use reduced significantly during long listening sessions.** The stream URL cache stores a slim copy of the yt-dlp info dict instead of the full raw object. A raw info dict for a YouTube video includes the complete format list (100+ entries), automatic captions for 30+ languages with per-segment timing data, thumbnail manifests, heatmap data, and internal yt-dlp bookkeeping fields. In Python memory this can be 10–50 MB per video. With a 120-entry cache that adds up to several GB. The new code strips all heavy fields (`formats`, `automatic_captions`, `thumbnails`, `subtitles`, `heatmap`, `comments`, etc.) before storing, keeping only the small metadata fields used by the player UI. The maximum cache footprint drops from several GB to a few MB.
