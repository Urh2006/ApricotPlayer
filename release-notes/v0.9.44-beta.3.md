# ApricotPlayer v0.9.44 Beta 3

Third beta in the 0.9.44 cycle. Addresses tester feedback from beta.2: automatic crashes during navigation, higher memory usage after recent betas, and volume resetting unexpectedly when closing the player screen with Escape.

## Bug fixes

- **Navigating results and menus no longer crashes the app.** Background worker results are delivered to the UI through a timer-driven queue (`process_queue`). The original drain loop wrapped both the queue read *and* the handler dispatch inside a single `try/except queue.Empty` block, so any unhandled exception thrown by a handler (for example, a `RuntimeError` raised by wxPython when a destroyed list control is accessed during rapid navigation) would escape all the way to wxPython's main-loop exception handler, which terminates the process. The queue read and the handler dispatch are now guarded by separate try/except blocks: `queue.Empty` from `get_nowait()` breaks the drain loop as before; any exception thrown inside a handler is caught and discarded, keeping the app running.

- **Volume no longer resets when closing the player screen with Escape.** `stop_player()` was unconditionally clearing `session_volume` whenever `reset_session=True`, which is the default. Because `back_to_results()` always calls `stop_player()` with the default arguments, pressing Escape (or navigating back to results while stopping playback) wiped whatever volume the user had set. Volume is now kept for the full app session and only disappears naturally when the app closes, matching the intended "session-based" behaviour described in the settings.

## Performance improvements

- **Memory use is lower after extended browsing sessions.** `metadata_hydration_urls` is a set used to avoid re-fetching yt-dlp metadata for the same result twice. It was never cleared, so after a long session with many searches it could accumulate thousands of URL strings. The set is now reset automatically once it exceeds 1 000 entries, which caps its steady-state size to a few tens of kilobytes. The practical effect is that a handful of previously-hydrated results may be re-fetched once per reset, which is negligible.
