# ApricotPlayer v0.9.44 Beta 8

Eighth beta in the 0.9.44 cycle. Fixes a bug where Alt+Tabbing to the player window while a screen reader (NVDA) was running would mute or heavily duck the system's main audio output for the entire duration the player had focus.

## Bug fixes

- **Alt+Tab to player mutes main speakers while the window has focus (NVDA audio ducking regression).** When NVDA is running with its audio-ducking feature enabled ("Duck when outputting speech and sounds"), Windows mutes all other audio whenever NVDA speaks. The player made this far worse than necessary: every call to `announce_player` (track started, volume changed, seek, etc.) called both `nvdaController_speakText` *and* then immediately fired a `wx.ACC_EVENT_SYSTEM_ALERT` WinEvent. NVDA monitors `EVENT_SYSTEM_ALERT` independently of the controller API and would pick up the WinEvent as a *second* speech request for the identical text — interrupting its own speech mid-sentence and restarting from the beginning. The result was that every announcement effectively doubled NVDA's speaking time, keeping audio ducking active far longer than expected, and users with "Mute all other sounds" configured saw their speakers appear permanently muted while the player window was in the foreground.

  Fix: `raise_accessibility_alert` now accepts a `skip_alert_event` flag. `speak_text` passes `skip_alert_event=True` whenever `nvdaController_speakText` returned success (return code 0). In that case `EVENT_SYSTEM_ALERT` is suppressed — NVDA already has the text; no second announcement is needed. The `EVENT_OBJECT_NAMECHANGE` and `EVENT_OBJECT_VALUECHANGE` WinEvents are still always fired so that JAWS, Narrator, and other screen readers that do not expose an nvdaController-style API continue to work correctly.
