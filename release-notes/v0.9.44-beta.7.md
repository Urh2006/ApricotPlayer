# ApricotPlayer v0.9.44 Beta 7

Seventh beta in the 0.9.44 cycle. Fixes a critical playback regression introduced in beta.6 that silently killed mpv on every launch when disk cache is enabled, restores stream compatibility reverted by beta.5, and corrects the keyboard hook dispatch order.

## Critical bug fixes

- **Player completely broken with disk cache enabled (regression from beta.6).** Beta.6 changed the mpv launch argument from `--demuxer-max-back-bytes` to `--demuxer-back-bytes`, believing the latter was the correct modern name. In fact, the bundled mpv 0.41 recognises only `--demuxer-max-back-bytes` and exits with *"option not found / Fatal error"* when given the other name. Because disk cache is enabled by default, every playback attempt since beta.6 silently failed at the subprocess level. Reverted to `--demuxer-max-back-bytes`.

- **Stream gets permanently stuck after network drop; does not recover when connection returns (regression from modular refactor).** mpv had no instruction to reconnect to an HTTP stream after a network drop. When the connection dropped, mpv exhausted its buffer, hit EOF, and stalled. When the connection returned, mpv simply stayed at the EOF position. Three reconnect options are now always passed to mpv for HTTP streams: `--stream-lavf-o=reconnect=1`, `--stream-lavf-o=reconnect_streamed=1`, and `--stream-lavf-o=reconnect_delay_max=5`. These tell libavformat to transparently retry the HTTP connection when it drops and to allow up to 5 seconds between reconnect attempts before surfacing an error.

## Other bug fixes

- **Stream loads for a few seconds then stalls (regression from beta.5).** Beta.5 changed the yt-dlp format selector from `best[ext=mp4]/best` to `bestaudio[ext=m4a]/bestaudio/…`. When this selector returns a DASH or HLS multi-track result, yt-dlp populates `requested_formats` instead of `url`. Beta.5 also added a fallback that read the URL from `requested_formats`. The problem: a DASH/HLS `requested_formats` entry contains a *segment manifest URL*, not a continuous stream URL. mpv receives one segment, plays a few seconds' worth of data, reaches the end of that segment, and stops. The format selector is reverted to `best[ext=mp4]/best` (the same selector used in v0.9.19) and the `requested_formats` fallback is removed. This restores the single continuous progressive stream URL that mpv can play indefinitely.

- **Background player Tab navigation broken (regression from modular refactor).** In the original code, `handle_background_player_tab_navigation` and `handle_player_tab_navigation` were called before checking whether the focused control was in the results list. The refactored `on_char_hook` reversed this order, so pressing Tab while results had focus triggered the results-list path and never reached the background player Tab handler. Order is now restored to match the original.
