# ApricotPlayer v0.9.44 Beta 2

Second beta in the 0.9.44 cycle. Addresses tester feedback from beta.1: navigation felt laggier than before the modular split, and checkboxes weren't responding correctly to Space.

## Bug fixes

- **Checkboxes now toggle correctly with Space.** When a checkbox had focus, pressing Space was falling through to the player shortcut layer and triggering play/pause instead of toggling the checkbox. This affected any checkbox outside the player screen while a background player was active. Fixed by letting `wx.CheckBox`, `wx.Slider`, and `wx.SpinCtrl` handle their native keys (Space, arrow keys) before the shortcut matching loop runs. Enter on player-screen checkboxes (repeat, bass boost, etc.) is intentionally unaffected — it still invokes the correct toggle handler.

## Performance improvements

- **Keyboard shortcut matching is dramatically faster.** `shortcut_matches()` is called more than 30 times per keypress in the central `on_char_hook` handler. Each call chained down to `shortcut_key_code()`, which was rebuilding a 30-entry alias dictionary and compiling two regex patterns from scratch on every invocation. All three objects are now module-level constants, built once at import time. The per-keypress regex overhead drops to near zero. This is the primary reason navigation felt sluggish — every Enter, arrow key, or Space triggered hundreds of dict constructions and regex compilations before anything visible happened.

- **`event_matches_shortcut` uses a pre-compiled split pattern.** The `|`-separator regex was also compiled fresh on each call. It is now a module-level compiled pattern alongside the other shortcut constants.

- **Startup activation reduced from 4 win32 calls to 2.** `activate_window_later()` was scheduling `foreground_window()` (which calls `AttachThreadInput` and several other win32 APIs) at four delays: 0 ms, 75 ms, 250 ms, and 750 ms. The 75 ms and 750 ms retries were added defensively but in practice added measurable overhead on slower machines right after launch. The default is now two delays: 0 ms and 250 ms. Tray-restore and post-update relaunch keep their own explicit delay lists and are unchanged.
